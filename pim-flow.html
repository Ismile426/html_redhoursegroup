Pimcore stores all five levels as rows in the same object tree. What separates a template from a variant is not a different table — it is the object's type flag (object vs variant) and whether inheritance is enabled.

2 · What the Pimcore object tree actually looks like

The admin "Data Objects" tree for range 40400

/Products 📁 Screws ← object folder (catalog) 📋 40400 Self-drilling facade DxL TX ← Parent Data Object (template) · fields: head_type, material, self_drilling=Y (inherited by all children) 🔩 404001 "4.8×25 mm TX20" ← Object variant (sellable) · overrides: drive=TX20, dia=4.8, length=25 🛒 404001-1-26-3-2 ← nested variant (add-on child) 🛒 404001-0-59-0-0 ← paint-only add-on 🔩 404002 "5.5×38 mm TX20 Premium"
A variant always sits under its parent. Move it out and it stops being a variant. This physical parent→child nesting is exactly how the wizard's live tree and the listing's Method-B tree are drawn.

3 · Inheritance — how attributes flow down

The single most important PIM concept for this model

Rule A child object shows the parent's value for any field it doesn't set itself. Set a value on the child and it overrides just that field. This is Pimcore attribute inheritance (allowInherit=true on the class).

🟠 Fixed on the template

head_type, material, self_drilling… set once on 40400. Every variant & sub-variant inherits them automatically — no re-typing, and a later edit on the parent cascades to all children.

🟣 Configurable on the variant

drive, major_diameter, length are empty on the parent and filled on each variant. That override is what makes 404001 a distinct SKU.

5 · How it actually works — a product manager in Pimcore

Building range 40400 and its SKUs, step by step in the admin UI

A · Create the template (parent)
  • 1In Data Objects, open the Screws folder → Add Object → Product. Name it 40400.
  • 2On the object, tick "Allow variants". Inheritance is already on for the class.
  • 3Fill the fixed fields once: Head Type = HWH, Material = Bi-metal A2, Self-drilling = Y, Washer = Y… These will flow to every child.
  • 4Set the allowed values for the configurable selects (Drive: TX20; Ø: 4.8/5.5; Length: 25/38). Leave them empty on the parent.
  • 5Attach relations: Brands (CREO, AXIS) and Cross-sell (EPDM washers, TX20 bit). Add Assets (photo, datasheet).
  • 6Save as Draft. The template is a spec — it is never sold.
B · Add a variant (the sellable SKU)
  • 1Right-click 40400 → Add Variant. Pimcore creates a child of type variant → key 404001.
  • 2Notice head_type/material are already filled — inherited, greyed. You don't touch them.
  • 3Set the variant name ("4.8×25 mm TX20") and the configurable picks: drive=TX20, Ø=4.8, length=25.
  • 4Narrow the brands to those this SKU actually sells under (e.g. CREO + AXIS ⊆ template brands).
  • 5The Calculated Value fields (weight, effective length) auto-fill from Ø/length.
  • 6Save & Publish. 404001 is now a live, sellable SKU.
C · Add a sub-variant (add-on)
  • 1Right-click 404001 → Add Variant again → a nested variant.
  • 2Set the four add-on axes: Configuration = WSH-ALU14B, Colour = RAL 8371 M, Size = 150, Style = RH4001 C25B.
  • 3The SKU key becomes 404001-1-26-3-2 (codes from each option). Brands narrow again (⊆ variant brands).
  • 4Everything else is still inherited from 404001 → 40400. Publish.
D · Quality gate & publish out
  • 1A Workflow moves each object Draft → In-Review → Approved. Only Approved+Published objects leave PIM.
  • 2Completeness check flags missing images/attributes before go-live.
  • 3DataHub (GraphQL) exposes published objects; the webshop reads them, and an export feeds Dynamics 365 F&O (product master + released variants).
  • 4F&O then owns price, stock & release-per-company — PIM owns the content & structure.

7 · How search works inside Pimcore

Why "red colour screw" finds the right object

Pimcore's backend search uses a MySQL InnoDB FULLTEXT index on the search_backend_data table — not Elasticsearch by default. When an object is saved, Pimcore concatenates its readable fields (name, input/select/textarea values, key) into that index.
Field typeIndexed for search?Note
Input / Textarea / WYSIWYG✅ yesFree text is fully searchable
Select / Multiselect✅ yes (the stored value)e.g. colour = "RAL 9005"
Numeric / Boolean✅ as textDiameter "4.8", self_drilling "Y"
Object relations (Brand, Cross-sell)⚠️ not by defaultStored as links, not text — needs a helper/index field
Inherited valuesDepends on configOften the child re-indexes inherited text
So a search for "red screw" matches because the colour select value and the product name/type are text-indexed. A pure relation (e.g. linked Brand object) won't match unless you add a searchable mirror field — a common reason a term "should match but doesn't". For large catalogs teams bolt on the Elasticsearch / Advanced Index bundle for faceted, typo-tolerant search on the storefront.
